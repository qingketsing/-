首先是RAG的整体概念分为六个部分，问题与动机，数据获取和知识库构建，切分与表示，检索与召回，重排与上下文构建，评估与优化。

其次是RAG主要解决的四个根本问题，知识不存在于知识库中，存在的知识无法使用，找到对应的知识较为困难，找到了对应知识也不一定可以用好。

使用RAG的主要解决的几个困难点：幻觉，准确，可拓展，成本，可控。

对于第二部分要学习
- **全量更新 vs 增量更新**
	- 全量更新
	 逻辑简单 一致性强 不容易遗漏变化 适合早期 demo 和小数据集。但是成本高 慢 数据越大越不现实 更新期间可能影响服务
	- 增量更新
	- 更快 更省算力  更适合大规模知识库 更适合频繁更新场景。但是逻辑复杂 容易漏更新 容易产生脏索引、重复 chunk、旧版本残留
- **文档变更检测**
	时间戳，哈希，版本号，结构化diff
- **删除旧版本**
	直接删除，版本共存检索时默认过滤旧版，软删除（标记inactive）
- **重建索引**
	向量索引 BM25 索引 混合检索的各类索引结构 元数据过滤索引
- **定时同步**
	按固定时间获取数据更新，内容变更，索引更新
- **热更新策略**
	后台构建新版本知识库或索引，完成后把查询流量切过去。或者新增内容先即时可检索，旧版本和脏数据稍后异步清理。
- **脏数据回滚**
	因为更新的数据不一定是永远正确的，可能会存在脏数据，所以需要回滚。回滚的策略有：基于版本回滚，基于原子事务日志回滚，标记删除和恢复回滚（参考删除旧版本的软删除）。

对于第三部分需要学习
- **切分的目标与作用**
	原有的内容太长，主题太杂，一个问题只和其中一小部分相关，整篇做向量表示会把局部信息“平均掉” ，检索返回整篇文档会带来大量噪声，而且上下文窗口有限，不能把整本资料都塞给模型 所以切分的核心目标不是“把文本切开”，而是：**把原始知识变成适合检索、适合命中、适合引用、适合被模型消费的基本单位。**
- **Chunk 粒度设计**
	这里的chunk粒度设计就是每个知识单元的大小和范围，常见的衡量方式有，字符数，token数，句子数，段落数，结构树。粒度设计是整个知识库的核心部分，因为它决定了可命中性和命中后信息的完整性。如果chunk过大，会导致一个chunk混有多个主题，向量表示也会变得模糊，上下文的噪声非常大，而且每条都会很长，浪费token。如果过小，会导致问题和答案分开，检索到了局部句子，但是没有完整的内容，模型无法回答，语义过于零碎，匹配容易出现错误。
- **切分策略**
	常见切分方式有按固定长度，按照段落，按照标题层级，按语义切分，分层切分。这些切分策略没有最优，只有最合适。一般切分策略如下
	- 小项目：固定长度 / 段落切分
	- 结构化文档：标题层级切分
	- 长文复杂材料：分层切分
	- 追求更高质量：结构 + 语义结合
- **Overlap 设计**
	Overlap 是相邻 chunk 之间保留一部分重叠内容。例如每个 chunk 300 token，前后重叠 50 token。
	
	为什么需要Overlap？
		语义经常会跨边界，单独检索某一个块不够完整。Overlap的作用就是通过保留局部连续性来减少残缺，解决边界缺失的问题。但是同样，Overlap的大小也要合适，太小会导致连续性不足仍然缺失信息，而太多则会导致检索时出现重复的区块和rerank难度上升，索引膨胀等问题。Overlap 不是“越多越保险”，而是：**在边界连续性和冗余成本之间做平衡。**
- **特殊内容的切分处理**
	FAQ最好作为整体保存，标题和正文整体保存，表格如果拆成文本会丢失原有格式，需要特殊处理，代码块应该和说明文本结合，法条需要保存层级和编号。
- **稀疏表示**
	稀疏表示主要是针对词项的表示方式，如 TF-IDF，BM25。稀疏是因为一个文本在整个词表空间里，只会命中很少一部分词，向量大多位置是 0，所以叫稀疏。它是对“精确词匹配”的一种强支撑。
- **稠密表示**
	稠密表示主要指 embedding，把一段文本映射成一个低维实数向量。和稀疏表示不同，它不是看显式词项，而是试图编码整体语义。工作原理是把chunk生成embedding，把用户的query做成embedding，然后通过各种相似度找出合适的chunk索引，返回chunk。它的主要应用场景是用户和文档“说法不一样”时，仍能基于语义把内容找回来。
- **混合表示**
	混合表示，就是把稀疏表示和稠密表示结合起来。稀疏负责关键词和精确匹配 稠密负责语义近似和表达泛化。常见形式有BM25 + embedding，稀疏检索、稠密检索并行，再做结果合并。这种查询方式更加稳定，回复更加准确，对复杂文档有较好的表现。
- **元数据设计**
	元数据就是附加在 chunk 上的结构化信息，不是正文内容本身，但对检索和管理非常重要。因为 RAG 不是只靠正文做检索。元数据能帮助： 过滤 排序 溯源 展示 版本控制 权限控制
- **切分与表示对检索效果的影响**
	检索效果并不只是 retriever 决定的。  用再好的 embedding 模型，再强的 reranker，如果前面的 chunk 单位和表示方式不对，整体效果还是会差。比如粒度不合适会导致召回粗糙，或者信息不完整，Overlap设置不合适会导致检索到了但是无法回答，表现方式只用稀疏或者只用稠密也会导致不稳定。

对于第四部分需要学习
- **检索与召回的目标与作用**
	检索与召回的目标，就是在海量文档、切片、知识片段中，**快速找出与当前查询最相关的一批候选结果**，供后续重排、上下文构造、生成回答使用。
	在完整 RAG 流程里，它承担的是**第一道筛选**，用于降低延迟和成本，降低模型幻觉，提高命中率，缩小搜索空间
- **查询理解（Query Understanding）**
	查询理解就是：系统先分析用户问题，搞清楚**用户到底在问什么**它会做意图识别、实体抽取、上下文补全和歧义消解。这样可以避免直接拿用户原句去检索造成召回偏移，尤其在多轮对话和企业知识库场景下特别重要。
- **查询改写与扩展（Query Rewrite / Expansion）**
	查询改写是把原始 query 变成**更适合检索的形式**。改写侧重把 query 变得更清晰、更结构化，扩展侧重补充同义词和相关概念提高召回率。但它也可能引入噪声，所以通常要控制扩展范围，或者采用多路召回而不是完全替代原查询。
- **稀疏检索方法**
	稀疏检索依赖关键词匹配和统计权重，代表方法是 BM25。它特别适合术语精确匹配、代码报错、产品名、接口名这类场景，因为这些场景里词面一致性很重要。它的优势是稳定、可解释、低成本，但对语义相近而字面不同的表达支持较弱。
- **稠密检索方法**
	稠密检索通过 embedding 把 query 和文档映射到同一向量空间，用向量相似度做匹配。它的优势是可以处理同义表达和语义相关问题，因此在知识问答中很常用。但它对精确关键词不够敏感，而且可解释性较弱，所以很多系统不会只用稠密检索，而是和稀疏检索结合。
- **混合检索方法**
	混合检索的核心思想是把关键词匹配能力和语义理解能力结合起来。常见做法有各取 Top-K，再融合；加权融合机制；分阶段策略。所以在实际项目里，混合检索往往比单一路召回更稳健，尤其适合企业知识库和复杂问答场景。
- **Top-K 召回机制**
	Top-K 召回的目的是在第一阶段尽量保证正确结果进入候选集合。K 过小会漏召回，K 过大又会引入噪声并增加重排成本，所以一般需要结合离线评估和线上延迟指标来调优。
- **多路召回策略**
	多路召回是为了提高第一阶段候选集合的覆盖率，本质上是从不同角度并行找答案。它可以基于不同检索算法、不同字段、不同 query 改写版本来做。核心收益是减少漏召回，核心难点是结果融合、去重以及成本控制。
- **检索结果融合**
	常见方法包括去重、加权打分和基于排名的 RRF。RRF是按不同检索器中的排名位置做融合，而不是直接比原始分数。这是很常见也很稳的方法。
- **召回质量评估**
	主要分为离线评估和在线评估。
	1）Recall@K：在前 K 个召回结果里，是否包含正确答案。  最核心。 
	2）Hit Rate / Hit@K ：前 K 里有没有命中至少一个正确结果。
	3）MRR（Mean Reciprocal Rank）：正确答案排得越靠前越好。
	4）nDCG：考虑排序位置和相关性等级，适合多相关结果场景。
- **召回失败类型分析**
	召回失败分析就是：当系统没找到正确内容时，去拆解到底失败在哪一层。召回失败应该分成几个层面看：查询理解、chunk 切分、索引更新、检索模型、结果融合和数据质量。先确认正确答案是否真的存在于知识库且已入索引，再检查 query 是否表达到了关键实体，再看 chunk 是否把答案切坏，最后分析检索打分和融合是否把有效结果压掉。
- **检索与召回优化策略**
	通过四层去优化：  
	第一层是 query 层，做改写、扩展和上下文补全；  
	第二层是数据层，优化 chunk 切分、元数据和索引更新；  
	第三层是检索层，结合 sparse、dense、多路召回和合理的 Top-K；  
	第四层是融合排序层，通过 RRF 或 reranker 提高候选质量。

第五部分
- **重排的目标与作用**
	重排的作用是在召回结果中进一步提升排序质量，把最相关、最有证据价值的内容排到前面。召回阶段更关注覆盖率，避免漏掉正确答案；重排阶段更关注精度，避免噪声进入上下文。
- **重排模型与重排策略**
	重排常见有规则式重排、Cross-Encoder 重排和基于 LLM 的重排。其中 Cross-Encoder 因为让 query 和候选文本进行深度交互，通常比 bi-encoder 相似度更准确，但推理成本更高，所以一般只对召回出来的小规模候选集做精排，而不是直接对全库检索。
- **候选结果筛选与过滤**
	重排后的候选并不等于最终可用上下文，还需要继续做筛选和过滤。常见做法包括按相关性阈值过滤、相似内容去重、清除低质量或过时内容，以及结合元数据做来源或类型限制。这样可以把有限上下文窗口优先留给真正有证据价值的内容。
- **上下文窗口分配**
	上下文窗口本质上是有限资源，所以必须做分配。在 RAG 中通常需要在系统指令、用户问题、历史对话和检索证据之间平衡 token 预算。一般做法是优先保证核心 query 和高相关证据，再根据场景动态决定是否保留更多历史上下文或补充证据。
- **上下文拼接与组织方式**
	上下文组织方式会直接影响模型对证据的理解和利用。常见做法是按相关性排序、按来源聚合或按任务结构组织，并给每段证据附上清晰标识，方便模型做引用和归因。
- **上下文压缩与去噪**
	上下文压缩就是在不丢失关键信息的前提下，**把证据压短**。去噪就是去掉和问题无关、重复、冗余或低质量内容。在有限 token 预算下保留尽量高密度的有效信息。常见方法包括抽取式保留关键句、摘要式浓缩内容，以及去重或者去除模板噪声（比如免责声明，固定模板）。
- **Prompt 构造与指令设计**
	Prompt 构造的核心是把任务目标、证据、约束和输出格式明确传达给模型。在 RAG 中，好的 prompt 不只是“让模型回答”，而是要明确要求它优先依据检索证据作答，在证据不足时说明不确定，并按指定格式输出。这能显著降低幻觉，提高答案可控性和一致性。
- **基于证据的生成（Grounded Generation）**
	Grounded Generation 的目标是让模型基于外部证据而不是纯参数记忆来作答。在 RAG 场景中，这能减少幻觉，并让答案具备可追溯性。主要做法就是通过提示词来控制边界。
- **引用与来源组织**
	引用与来源组织的核心作用是提升答案的可验证性和可信度。这不仅方便用户核查，也方便开发者做错误定位和效果评估。
- **生成结果质量控制**
	生成结果质量控制通常包括格式约束、事实一致性检查、引用校验、拒答策略和必要的后处理过滤。因为即使检索到了正确证据，模型在生成时仍可能误解、遗漏或过度推断，所以输出层还需要做额外控制。这部分对于稳定性和上线性非常关键。
- **重排与生成阶段的常见失败类型**
	常见失败可以分成三层：  
	第一是重排失败，导致真正相关内容没进入前列；  第二是上下文构造失败，比如组织混乱、重复过多或压缩失真；  第三是生成失败，比如幻觉、证据利用不足、引用错位或不遵循指令。  
	所以“检索到了但没答好”并不一定是检索问题，也可能是后面的重排、上下文构造或 prompt 设计出了问题。
- **重排、上下文构造与生成的优化策略**
	这一阶段的优化重点不是单纯“让模型更会说”，而是让证据更好地进入模型并被正确使用。主要可以从四个方面进行优化：
	第一层是重排，提升候选排序质量并做去重；  
	第二层是上下文构造，优化窗口分配、组织方式和压缩策略；  
	第三层是 prompt 设计，明确 grounded 约束和输出格式；  
	第四层是生成后质量控制，包括事实校验、引用校验和拒答机制。

第六部分
- **检索阶段评估指标**
	检索阶段主要关注候选集合是否覆盖正确答案，所以最核心的是 Recall@K 和 Hit@K。如果正确文档没有进入候选集，后续重排和生成就没有基础。  此外也可以结合 MRR 和 nDCG 看排序质量，用 Precision@K 辅助分析噪声比例。
- **生成阶段评估指标**
	生成阶段评估关注的是答案本身的质量，通常会看正确性、完整性、流畅性、一致性和格式符合度。它比检索评估更难，因为同一个问题可能有多种正确表达，不能只做字符串匹配。所以实际项目里通常会结合人工评审、规则校验和 LLM 评估一起做。
- **端到端评估体系**
	端到端评估关注的是从用户提问到最终答复这一整条链路的综合表现，比如回答正确率、任务完成率和用户满意度。工程上通常会把端到端评估和分层评估结合起来：端到端看整体，分层评估看问题出处。
- **答案真实性与依据一致性评估**
	这两个是答案质量的重要指标，真实性关注答案事实是否正确，依据一致性关注答案是否真的来源于提供的上下文，而不是模型自行补全。
- **相关性与可用性评估**
	相关性和可用性评估关注的是答案是否真正解决了用户问题。主要看答案是否相关、是否具体、是否可用。
- **延迟与成本评估**
	因为多路召回、重排和多次模型调用虽然可能提升效果，但也会显著增加响应时间和单次请求成本。所以工程上要在效果、延迟和成本之间做平衡，例如通过控制 Top-K、减少不必要链路、缓存高频结果等方式优化。
- **错误案例分析（Bad Case Analysis）**
	错误案例分析就是指分析它**为什么错**、**错在哪**、**能怎么改**。主要分析是检索未命中、重排错误、上下文截断还是生成幻觉，再统计高频错误类型。
- **分层问题定位方法**
	一般会按链路分层定位：先确认答案是否存在于知识库并已建索引，再看检索有没有召回正确 chunk，然后看重排是否把它排到前面，再检查它是否进入最终上下文，最后分析模型是否正确利用了证据。  这样可以把一个笼统的“回答错误”拆成可操作的定位流程，避免盲目修改 prompt 或模型参数。
- **离线评测与在线评测**
	离线评测更适合做可重复、低成本的版本比较，比如比较不同检索器、不同重排模型或不同 prompt；在线评测则更能反映真实用户分布和实际业务价值。一般做法是先用离线评测筛选方案，再通过在线评测验证真实效果，二者结合才能形成完整评估体系。
- **A/B 测试与迭代优化**
	A/B 测试的作用是验证某个改动在真实流量下是否真的带来收益。这样优化不再是基于猜测，而是基于真实用户反馈和线上数据做决策。
- **系统监控与反馈闭环**
	系统监控与反馈闭环的目标是让 RAG 在上线后持续可观测、可迭代。这样系统才能形成“上线—监控—发现问题—修复—再验证”的持续改进闭环。


1. 什么是RAG，为什么要做RAG (1)
	RAG是检索增强生成，通过检索本地数据库的内容，然后嵌入到提示词中，提高LLM生成内容的准确率。如果不使用RAG，可能会存在幻觉，私有知识不存在，微调成本高，知识过时等问题，所以需要检索增强生成来让LLM回复的内容更符合我们的要求。

2. RAG工作原理 (1)
	简单来说就是先检索资料然后让LLM去根据材料回答问题。基本的流程可以分为五步，准备知识库，获取用户问题，检索相关资料，将问题和资料提交给大模型，返回答案。

3. RAG优点，常见应用场景 (2)
	它不仅可以回答问题，而且可以根据现有材料准确的回答问题。降低了幻觉，更容易给出回答的根据，而且可以降低成本，更适合私有化场景，可控性更强。

4. RAG工作的流程 (1-5)
	获取数据->数据清洗->chunk切分->建立索引->用户提问->检索召回->重排->上下文拼接
	->LLM生成->评估结果

5. RAG的不同分类（通过检索方式）(2,4)
	1. 关键词检索型：通过对关键词进行检索，一般使用TF-IDF或者BM25等检索算法进行检索。优点是对关键词敏感，在日志检索和FAQ这种场景非常合适。缺点是对于同义表达、意图相近但词不一样时容易漏召回。
	2. 向量检索型：通过将chunk计算成一个向量，然后将问题转化成向量，通过向量距离找出最相近的内容。一般使用余弦相似度，欧几里得距离，点积来进行检索。优点是能够通过语义进行检索，更贴近自然语言的表达，需要理解意思而不是单纯的针对词语进行的检索。缺点是对精确术语、数字、代码、缩写可能不稳定，依赖embedding的准确度。
	3. 混合检索型：同时结合关键词检索和向量检索，再合并检索结果。检索方法有- BM25 + Embedding ，Sparse + Dense Fusion，Hybrid Search。这是现在最主流的一类。优点是可以兼顾语义匹配和关键词匹配，比单一的检索方式更加稳定，对于复杂问题有更强的处理能力。缺点则是系统复杂，需要结合两种检索方式的结果。
