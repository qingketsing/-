1. GMP模型
	GMP模型指的是Goroutine,Machine,Processor。分别是协程，线程，处理器。每一个goroutine创建后会放入P的本地队列，然后分配给Machine。machine可以对自己所拥有的goroutine的进行微观层面的轮流处理。当machine的负载不均衡的时候，P会主动的从有多个的另一个P中"窃取"任务。保证系统的高利用率。M是用来处理goroutine的单位，goroutine只有绑定了M之后才能进行工作，P的主要职责是用来调度goroutine。
2. channel原理
	channel的数据结构主要包含以下内容：
		buff：缓冲区，用于存储通道内的数据
		qcount：当前缓冲区中内容的数量
		dataqsize：缓冲区的大小，也就是通道的容量
		sendx：下一个发送的索引
		recvx：下一个要接收的索引
		recvq：接收队列
		sendq：发送队列
		lock：互斥锁，用于保护chan的并发访问
	对于发送和接受操作，都会经过四个过程：检查，尝试直接发送，等待，唤醒
		
		对于发送操作而言：
			1. 检查通道状态，如果channel关闭，那么这个发送操作会导致panic
			2. 尝试直接发送，如果有可用的缓冲区，那么就会直接发送
			3. 如果没有可用的缓冲区，就会阻塞并且等待接收者从缓冲区中接收。
			4. 如果有等待接收的Goroutine，就会直接唤醒并发送。
		
		对于接收操作而言：
			1. 如果channel已经关闭而且缓存区为空，那么就会返回零值和已关闭的标志
			2. 尝试直接接收，如果有可用的缓冲区就直接接收
			3. 如果没有可用的缓冲区，则接收者阻塞并等待发送者发送到缓冲区。
			4. 如果有等待发送的Goroutine，就会直接唤醒并接收。
3. GC（标记清除法，三色回收）
	标记清除是指先遍历所有内存，然后进行标记，标记结束后再进行清除，这个过程是串行的，不是并行的。三色标记则是标记的一部分，在三色标记这个标记过程中是并行而且并发的，可以同时扫描并且标记多个进程，标记黑白灰，在标记结束后进行清理。
4. Interface的原理
	`interface` 是一种强大的抽象机制，用于定义一组方法签名，而不提供具体的实现。它是一种抽象的类型，通过接口，Go 语言支持多态等面向对象的编程概念。
	interface同时有两种不同的处理，一种是非空接口，也就是包含函数的接口，iface，另一种则是不包含函数的接口，也就是空接口，eface。
5. go runtime
	runtime是go的运行时系统，是go语言的一个核心部分，负责管理的部分有goroutine调度，GC，内存分配和系统调用等操作。具体细节见上面内容和第七章内存管理。带一嘴三种内存对象的大小以及在哪里分配。
6. mutex
	mutex就是Go的互斥锁，主要有两个功能，就是上锁和解锁，上锁后，其他的goroutine无法同时访问资源，只有解锁后才能访问。
	
	锁根据使用场景可以分为两种，
	一种是悲观锁，是为了防止冲突将频繁发生，通过加锁来防止冲突，所有操作都需要等解锁后才能进行的。通常直接通过 `sync.Mutex` 或 `sync.RWMutex`。使用场景多为数据竞态严重的场景，比如数据库的写操作这种。
	
	一种是乐观锁，只有在提交或修改数据时，才会检查是否有其他 Goroutine 修改了数据。如果有冲突，就回滚操作并重试。经常使用`sync.atomic`来实现，使用场景多为并发冲突较少或数据读取操作多，写入操作少的场景。
7. slice
	在 Go 语言中，`slice` 是一个结构体，包含三个字段：
	    ptr `*`array  // 指向底层数组的指针
	    len int     // 当前 slice 的长度
	    cap int     // 当前 slice 的容量

# 7.1内存分配器

### 分配方法

1. **线性分配器**：维护一个指针，当收到内存申请的时候检查空间大小，然后返回分配内存区域，并修改指针位置。
	- 无法在内存释放时复用内存
	- 需要配合合适的垃圾收集算法来实现，比如标记压缩，复制回收，分代回收等。
2. **空闲链表分配器**：可以复用已释放的内存，在其内部维护一个类似链表的数据结构，当用户程序申请内存时，空闲链表分配器会依次遍历空闲内存块, 找到足够大的内存，然后申请新资源并修改链表。但是因为需要遍历，所以复杂度为O(n)。这种分配器有常见的四种区域选择策略
	- 首次适应：每次都从链表头部开始遍历
	- 循环首次适应：每次都从上一次结束的位置开始遍历
	- 最优适配：遍历整个链表，找到最优位置并返回
	- 隔离适配：将内存分割成不同的链表，每个链表内部的块的大小相同，申请内存时优先找到合适的链表，然后再在这个链表中找到合适的位置。
- Go语言使用的内存分配策略与第4种策略有些相似。

### 分级分配

这里的分级分配制的是对象的空间分配。空间分配依靠两个分级，一个是按照离CPU的距离，一个是按照对象的大小。Go把对象分为了三个尺寸，并且提供了不同的分配策略：
	1. 微对象：0-16B 哪里有空塞在哪里
	2. 小对象：16B-32KB 从线程缓存，到中心缓存，再到页堆逐级分配
	3. 大对象：32KB-+∞ 直接选择页堆进行分配

这里注意一下，在Go源码和各类书籍的翻译中，mcache等同thread cache，mcenter等同于center cache ，page heap等同于mheap。

### 虚拟内存布局

注意，这里的虚拟内存和真实的内存大小并不相同，比如我们可以有512G的虚拟内存，但是我们的运行内存可能只有16G，这两者是不冲突的。因为虚拟内存更像是门牌号，有点类似支持的物理内存的上限。

#### 线性内存

在线性内存中，有三个部分，spans，bitmap，arena。这三者的虚拟内存分别为512MB，16GB，512GB。当Go语言创建一个对象的时候，首先会在arena区域分配一个空间，然后根据这个空间的地址对bitmap进行标记，spans则是根据索引，对用来管理这个分配的空间的对象进行标记。spans中的元素为mspan，是Go内存管理的基本单位。以下是对这个单位的描述，但是注意一点，这个mspan不只是针对线性内存而设定的，在后续的稀疏内存中也是如此。
```Go
type mspan struct {
    next *mspan       // 下一个span（链表指针）
    prev *mspan       // 上一个span
	// ... 此处省略一些元素，不代表只有这么多参数
    allocBits *gcBits   // 分配位图
    gcmarkBits *gcBits  // GC标记位图
}
```

这样的虚拟内存分配办法存在许多问题，首先就是会和C语言冲突，并不是按需分配的。为了解决上述问题，Go语言在更新后使用了稀疏内存的处理办法。

#### 稀疏内存

稀疏内存实现了按需分配，并且将可用的虚拟内存拓展到了256TB（4,194,304 (二维大小) × 64MB (每个arena的大小)），使用的办法是设置一个二维数组管理所有内存，每个单元都会管理64MB的内存。创建对象的过程如下：
```
创建对象 → 分配Page → 计算地址 → 定位heapArena → 绑定关系
```

这里我们提一嘴，为什么二维大小是4194304：

```
// 64位地址的结构
63              48|47              0
[ 未使用(16位) ][  48位可用地址空间 ]

// 这48位地址又被分成：
[ arena索引(39位) ][ 页内偏移(9位) ]

// arena索引的39位进一步分成：
[ 一维索引(17位) ][ 二维索引(22位) ]
// 17 + 22 = 39位

二维索引占22位 → 2^22 = 4,194,304
```

### 地址空间

四种内存状态：

```
None      → 还没碰的内存
Reserved  → 预留了门牌号，但不能住人
Prepared  → 装修好了，随时可以入住
Ready     → 正在使用中
```

这四种内存状态的转换靠的是内存状态切换函数：
```
sysAlloc    // 找房东租新房
sysFree     // 退租
sysReserve  // 先占个位置，以后用
sysMap      // 准备入住
sysUsed     // 开始使用
sysUnused   // 暂时不用了，但别给别人
sysFault    // 出问题了，封锁现场
```

## 内存管理组件

Go语言的内存分配器包含几个重要组件：内存管理单元、线程缓存、中心缓存和页堆。这几个重要组件对应的数据结构是runtime.mspan, runtime.mcache, runtime.mcentral和 runtime .mheap。这里也就是我们前面分级分配讲到的一些数据结构。

### 内存管理单元

runtime.mspan是Go语言内存管理的基本单元，也就是我们在前面说到的mspan。这里我们再次列出mspan的struct，并进行参数的解释。

``` Go
type mspan struct {
    next *mspan       // 下一个span（链表指针）
    prev *mspan       // 上一个span
    startAddr uintptr // 管理的起始地址
    npages uintptr    // 管理多少页（每页8KB）
    
    state       mSpanStateBox // 当前状态（空闲/使用中/死亡等）
    spanclass spanClass // 跨度类（管理多大对象）
    
    elemsize uintptr    // 对象大小
    nelems uintptr      // 对象数量
    freeindex uintptr   // 下一个空闲对象的索引 用于快速查找
    
    allocBits *gcBits   // 每个bit对应一个对象，1=在用，0=空闲
    gcmarkBits *gcBits   // GC时标记存活对象，1=存活，0=可回收
    
    // ... 还有其他字段，但核心是这些
}
```

这里我们注意到有个状态state，我们再简单列出状态转移规则：
```
        分配时              GC清除时
mSpanFree ──────→ mSpanInUse ───────→ mSpanFree
         │              │
         │ 手动分配      │ GC标记阶段
         ↓              ↓
      mSpanManual   不能转换！
```

所谓的跨度类spanclass是一个8位的int，这个8bit的int的前七位是用来获得对象类型的id，而这个对象类型就是内存占用的大小所对应的对象，简单来说就是用来区分对象的大小的，最后一位则是判断这个对象是不是指针
```
spanclass spanClass  // 其实是uint8，但用类型区分
// 编码方式：
// 前7位：跨度类ID (0-67)
// 最后1位：是否包含指针(1=不扫描,不是指针,0=要扫描,是指针)
// 例如：
spanclass = 5<<1 | 0  
// = 管理48字节对象(第5类) + 需要GC扫描(含指针)
spanclass = 5<<1 | 1  
// = 管理48字节对象 + 不需要GC扫描(不含指针)

spanclass (8位二进制)
┌─────────────────┐───────────┐
│ 7位class ID     │ 1位noscan │
│ (0-127)         │ (0或1)    │
└─────────────────┴───────────┘
         ↓              ↓
    决定对象大小    决定是否要GC扫描
```

### 线程缓存

runtime.mcache是Go中的线程缓存，与线程上处理器一一绑定，用来缓存微小对象。每一个线程缓存都有68 * 2个runtime.mspan对象。它有三个主要操作：初始化，替换，微分配器。

#### 初始化

操作流程为：从堆里获得空的内存，然后进行分配对象，设置为空的mcache对象，随后与处理器绑定。然后这个 mcache 会和某个处理器绑定，之后这个处理器上的 goroutine 分配小对象时，就直接从这个 mcache 拿，不需要加锁。

#### 替换（或者可以叫refill）

个人更倾向于refill来解释，而不是替换，由于它的操作执行逻辑是如果mspan用完了，那么就会从mcentral里拿取新的mspan，放到已经用完的mcache中。（这种操作更像是续杯，而不是替换）。需要注意的是，如果mcentral也没有空余的空间，那么`mcentral.grow()` 会调用 `mheap.alloc()`，从操作系统拿新的内存页，切成 mspan 再返回。refill之后旧的mspan对象则是交给mcentral管理和GC清扫。这样的话就实现了分配和GC两个系统的独立，mspan是流动的，而GC则是只跟踪对象。

#### 微分配器

微分配器包含了三个参数，tiny，tinyoffset，local_tinyallocs。具体作用见注释部分。

```Go
type mcache struct {
    tiny             uintptr     // 指向堆中的一块内存（当前正在用的微对象块）
    tinyoffset       uintptr     // 这块内存里，下一个空闲位置的偏移量
    local_tinyallocs uintptr     // 统计：这个 mcache 分配了多少个微对象
}
```

### 中心缓存

首先我们要知道中心缓存的定位，它更像是Go内存分配中的中枢神经系统。

```
线程缓存 (mcache)  ← 无锁，每个P独享
    ↓ 当本地不够时，通过 refill 来拿
中心缓存 (mcentral) ← 需要加锁，全局共享
    ↓ 当中心不够时，通过 grow 来拿
堆 (mheap) ← 从操作系统申请内存
```

它是 mcache 和 mheap 之间的缓冲层，管理着同一种规格的所有 mspan。

mcentral同样是有自己的结构体的。
```Go
type mcentral struct {
    spanclass spanClass          // 这个中心缓存管什么规格（比如16字节）
    partial   [2]spanSet         // 有可用空间的 mspan
    full      [2]spanSet         // 没有可用空间的 mspan
}
```

为什么 partial 和 full 都是 [2]spanSet？这是为了支持并发 GC 的设计：

| 下标           | 含义                           |
| ------------ | ---------------------------- |
| `partial[0]` | 已经**被 GC 清理过**的、有空闲空间的 mspan |
| `partial[1]` | **还没被 GC 清理**的、有空闲空间的 mspan  |
| `full[0]`    | 已经**被 GC 清理过**的、已满的 mspan    |
| `full[1]`    | **还没被 GC 清理**的、已满的 mspan     |

#### 核心函数：cacheSpan()

这个函数的作用是找到一个可用的 mspan 返回给 mcache。这里我们把它的步骤简化介绍一下：
```
1. 从 partial[0] 拿 → 有就直接用 // 这些是已经清理过的，拿来就用，不用等。
2. 从 partial[1] 拿 → 拿到后清理一下再用 // 这些是有空间但没清理的，扫掉死对象就能用。
3. 从 full[1] 拿 → 清理后如果有空位就用，没有就放回去 // 这些是之前满的，清理后可能腾出空间
4. 最后实在没有 → grow() → 从堆申请新内存
5. 更新alloCache
   allocCache是 mspan 里的一个位图缓存，用来快速找到空闲对象。这里就是把它更新到当前空闲位置
```

这里我们再大体介绍一下grow( )函数。
```
调用方 (mcentral.grow) 
    ↓ 需要 npages 内存
mheap.grow(npages)
    ↓
[预留区] → 有就直接用
    ↓ 没有
[操作系统] → sysReserve + sysMap
    ↓
[初始化] → 创建 heapArena，注册到 arenas
    ↓
返回新内存地址给调用方
```

## 内存分配
