1. RabbitMQ是什么，作用是什么，结构是怎样的？
	RabbitMQ是一个消息队列中间件，作用是将短时间内的大量消息异步处理，实现削峰填谷的作用，避免短时间高并发导致的服务宕机。结构分为生产者，Broker，消费者。Broker与其他两个通过Connection连接，Connection中有通道Channel。Broker自身包含Vhost，Vhost中包含着多个Exchange和Queue，Exchange通过Binding与Queue连接。
2. 什么是AMQP协议？
	这是一个高级的消息中间件的通信协议。就像是HTTP是web的协议，MySQL是数据库的通信协议一样。AMQP是消息中间件的通信协议。主要规定了消息怎么发，怎么路由，怎么收，怎么确认消息等。
3. 生产者，消费者概念？
	生产者就是发出消息的一端，消费者则是用来处理消息的一端。
4. 什么是交换机，以及四种类型
	交换机是用于将消息发送到合适的队列的一个结构。四种类型是Direct，Fanout，Topic，headers。Direct是指routing key == binding key的时候才会发送到这个队列里，Fanout则是通过Binding直接发送向所有绑定的队列，可以发送向多个，Topic是模糊匹配，通过通配符进行判断是不是应该发送到这个队列，headers则是通过对headers中的参数进行一系列参数判断是否可以发送到这个队列。
5. 什么是消息确认机制
	消息确认机制的英文缩写是ack，作用是返回确认，通过一系列办法可以借助ack实现避免消息的重复发送和消费。
6. 什么是持久化
	持久化就是将内存中的内容存储到磁盘中，防止宕机后的信息丢失。但是对于持久化也有前提条件，主要是三个前提条件，一是delivery code = 2，二是设置交换机可持久化，三是队列设置持久化。
7. Vhost是什么？作用是什么
	Vhost是在Broker层上创建的虚拟化环境，作用是防止其他进程对MQ产生影响，类似创建一个沙箱避免外部对其内部的影响。
8. 什么是手动 ack？它解决了哪些问题？
	手动ack并不是指人工去确认，而是相对于自动ack而言的。自动ack是只要收到消息就会返回，而手动ack则是任务结束后才会发出ack，至于什么时候发出，则由收到消息的一方决定。
9. 什么情况下会出现重复消费？对于消费者如何处理重复消息？
	两种情况，一种是生产端重复发送，一种是队列重复发送到了消费者端。消费者对于重复消息的处理办法比较复杂，主要就是消息要有全局唯一ID，消费者手动Ack，保证业务的幂等性（执行一次和执行n次的效果一样，实现办法有把业务当作状态机来处理，还有就是唯一性的处理，第二次之后的全部无效），使用数据库的唯一键（不能使用redis，因为redis会有TTL），失败有限重试（多次失败设置为死信）。
10. 什么是死信队列，什么时候会使用？
	死信队列是指用来处理各种异常的消息，比如超时未消费，超出队列长度，消息被拒而且没有重新入队的信息。
11. 如何处理消息消费失败？（nack、requeue、死信队列）
	这里主要是区分临时失败还是永久失败，对于临时失败返回的nack/requeue= true，然后消息进入重试队列。对于永久失败则是nack/requeue = false，进入死信队列。但是生产环境中一般不推荐无限requeue
12. 消息持久化和队列持久化的区别？
	队列持久化是指将队列的元信息和各种参数进行持久化保存，但是队列中的消息不会保存。而消息持久化保存的是消息体本身，也就是生产者发送的消息本体。
13. RabbitMQ 集群和镜像队列是什么
	RabbitMQ集群中的节点是对等的，客户端可以向任意一个节点发送消息而保证结果正确，镜像队列，每个节点的队列和exchange不一定是一致的。而镜像队列则是所有队列都有各自的队列，这样就不再是单点的，哪怕挂了一个也可以提升某个副本继续提供服务从而保证高可用性。镜像队列的各个节点并非对等，而且是有主从关系的。
14. RabbitMQ 与 Kafka 在性能和场景上的差异？
	RabbitMQ更倾向于任务队列的模型，强调灵活的路由和消息确认。而Kafka偏向分布式事件流平台，强调高吞吐和大数据日志流。一个是消息代理，一个是事件流平台，RabbitMQ更适合低延迟的业务消息，灵活路由，消费确认的任务场景，而Kafka更擅长大量数据的吞吐和海量数据的保留。对于业务异步通知更适合RabbitMQ，而大数据的日志流和海量数据更适合Kafka。
