### 5. gateway、MDS、datanode、PostgreSQL 分别负责什么

- gateway：对外 HTTP 入口，负责上传、下载、目录和文件查询
- MDS：元数据服务，负责文件、目录、chunk、replica、节点信息管理
- datanode：真正存放 chunk 数据的节点
- PostgreSQL：持久化 MDS 元数据

### 6. 你说“打通核心组件链路”，具体链路是什么

核心链路是：
用户 -> gateway -> MDS -> datanode -> Postgr eSQL
其中数据写入 datanode，元数据写入 PostgreSQL，调度和状态管理由 MDS 完成。

### 7. 上传、下载、删除流程分别是什么

- 上传：gateway 收请求 -> 向 MDS 创建文件/会话 -> 获取上传目标 -> 写入 datanode -> 提交 chunk 元数据 -> 完成并校验上传
- 下载：gateway 向 MDS 获取下载计划 -> 按 chunk 从 datanode 拉取 -> 返回给客户端
- 删除：gateway 查文件 chunk 和副本 -> 通知 datanode 删除数据 -> 调用 MDS 删除元数据

## K8s / StatefulSet / PVC

### 1. 为什么 datanode 要改成 StatefulSet

因为 datanode 存真实数据，不能像无状态服务那样随便重建后丢数据。

### 2. Deployment 和 StatefulSet 有什么区别

- Deployment：适合无状态服务，Pod 名字和存储都不稳定
- StatefulSet：适合有状态服务，Pod 身份、网络名、卷绑定都稳定

### 3. datanode 为什么是有状态服务

因为它本地保存 chunk 数据，Pod 重启后需要继续使用原来的数据目录。

### 4. 为什么 datanode 需要 PVC

为了让数据目录不跟 Pod 生命周期一起消失，模拟节点本地数据持久化。

### 5. PVC 和 emptyDir 有什么区别

- emptyDir：Pod 删除/重建就丢
- PVC：卷独立于 Pod，重建后还能挂回

### 6. StatefulSet + PVC 如何保证数据持久化

StatefulSet 给每个 Pod 固定身份，PVC 给每个 Pod 固定卷；Pod 重启后会重新挂回原来的那块卷。

### 7. Headless Service 有什么作用

给 StatefulSet 提供稳定的 Pod DNS 身份，便于每个 Pod 有独立网络标识。

### 8. 普通 Service 和 Headless Service 有什么区别

- 普通 Service：统一入口、负载均衡
- Headless Service：不做统一 VIP，直接暴露 Pod 级 DNS

### 9. 如果 datanode Pod 重启，数据如何恢复

K8s 会重建 Pod，StatefulSet 会把同名 PVC 重新挂载到 /data/datanode，数据继续可用。

### 10. 如果扩展成多副本 datanode，需要改什么

需要补：

- 多副本 StatefulSet
- 每 Pod 独立 advertise URL
- MDS 更真实的节点选择和副本分配
- gateway 不再假设单一 datanode 入口

### 11. 你怎么排查 CrashLoopBackOff

一般顺序是：

- kubectl get pods
- kubectl describe pod
- kubectl logs
- 看依赖是否 ready
- 看镜像、env、探针、端口、存储挂载是否有问题

### 12. 你怎么排查 Service 访问不通

重点查：

- Service selector 是否匹配 Pod
- Endpoints 是否生成
- 端口和 targetPort 是否一致
- Pod 是否 ready
- DNS 和网络策略是否有问题

### 13. minikube 和真实 K8s 集群有什么区别

minikube 更像本地开发环境：

- 单节点居多
- 资源少
- 本地镜像需要手工 image load
- 网络、存储、调度都比真实集群简单很多

## smoke test / Metrics

### 1. scripts/poc-smoke.sh 是做什么的

它是 PoC 端到端冒烟脚本，用来自动验证核心链路是否可用。

### 2. 为什么需要冒烟测试

因为“Pod 都 Running”不等于业务链路真的通。冒烟测试验证的是“系统是否真的能用”。

### 3. 冒烟测试覆盖哪些流程

覆盖：

- 健康检查
- 上传小文件
- 查询元数据
- 下载文件
- 内容一致性校验
- 删除文件
- 删除后二次查询确认

### 4. 内容一致性怎么校验

把上传前原始内容和下载后的内容做字节级比对，脚本里是直接比较文件内容。

### 5. 冒烟测试失败时如何定位问题

先看失败停在哪一步，再顺着链路排：

- healthz 失败：先查 gateway 和上游依赖
- upload 失败：查 gateway -> MDS -> datanode
- download 失败：查下载计划和 datanode 数据
- delete 失败：查 chunk 元数据和 datanode 删除闭环

### 6. 冒烟测试能否接入 CI

能，但更适合作为 integration/e2e job，不应该混进默认 go test ./...。

### 7. 冒烟测试能证明生产可用吗

不能。它只能证明“最小核心链路可用”，不能证明高并发、故障、扩容、HA、性能都没问题。

### 8. Metrics 具体暴露了哪些指标

当前主要是服务级指标：

- HTTP / RPC 请求数
- 错误数
- 延迟
- gateway 上传下载相关指标
- MDS RPC 指标
- datanode 数据面指标

### 9. Prometheus 如何采集指标

服务暴露 /metrics，Prometheus 通过 scrape 或 ServiceMonitor 定期拉取。

### 10. Alertmanager 负责什么

负责接收 Prometheus 触发的告警，做分组、抑制、路由和通知分发。

### 11. 健康检查和 Metrics 的区别是什么

- 健康检查：回答“服务现在活着吗”
- Metrics：回答“服务现在运行得怎么样、趋势如何”

### 12. AI Agent 服务应该监控哪些指标

重点看：

- 请求量、成功率、错误率
- P95/P99 延迟
- Token / 调用成本
- 外部依赖错误率
- 队列积压
- 内存和 CPU
- 超时、重试、限流情况

## content_base64

### 1. 为什么当前上传接口使用 content_base64

因为 PoC 阶段要尽快打通最小链路，content_base64 最容易演示和调试。

### 2. 为什么它只适合 PoC

因为它简单，但不适合大文件和高并发，不是正式上传方案。

### 3. base64 上传大文件有什么问题

主要问题：

- 体积膨胀约 33%
- gateway 需要把整块内容吃进内存
- 传输和解析成本高
- 不适合断点续传和流式写入

### 4. 正式大文件上传应该怎么设计

更合理的是：

- 分片上传
- 流式传输
- 会话化上传
- 每片独立校验
- 最后由 MDS 提交元数据

### 5. 如何避免 gateway 内存被打爆

关键做法：

- 不把完整文件一次性读入内存
- 改成流式处理
- 分 chunk 边读边写
- 限制单请求大小和并发
- 必要时做背压和限流

### 6. 为什么 PoC 文档要明确限制

因为要防止别人把“演示接口”误当成“正式设计”，提前把边界讲清楚比后面补锅更专业。
