# DND AI BOT 项目面试问答

## 1. 请介绍一下 DND AI BOT 项目

DND AI BOT 是一个面向 DND 跑团场景的 AI 后端系统，我用 `Golang + PostgreSQL + Redis + RabbitMQ + RAG + Agent Runtime` 做了一套支持多轮对话、状态持久化和异步处理的服务。

它的核心目标不是做一个普通聊天机器人，而是做一个能长期维护游戏上下文的 AI DM 基础设施。系统支持用户注册登录、多会话聊天、规则/设定检索、角色和游戏状态持久化、战斗状态管理，以及异步消息任务追踪。

从本质上说，这个项目是在解决“如何让大模型在一个强规则、强状态、长上下文的游戏场景里稳定工作”的问题。

## 2. 这个项目解决了什么问题？

它主要解决了三类问题。

第一类是**上下文和状态不连续**。普通聊天模型很难稳定维护长链路剧情、角色信息、任务进度和战斗状态，所以我把这些内容结构化存到数据库里，包括 `session`、`game_state`、`encounter`、`session_memory`。

第二类是**规则和设定不可靠**。DND 规则很多，直接靠模型记忆容易幻觉，所以我做了 `rules/lore` 双知识库的 RAG 检索，把规则和设定先查出来，再交给模型使用。

第三类是**复杂请求慢且不稳定**。一个请求可能要多轮推理和工具调用，所以我设计了 `message_jobs + outbox_events + RabbitMQ` 的异步消息链路，用来支持可靠接单、任务追踪和后续 worker 执行。

## 3. 为什么 DND 跑团场景需要 AI Agent？

因为 DND 不是普通闲聊，它是一个**强上下文、强状态、强规则依赖**的场景。

比如玩家说“我攻击最近的敌人”，系统不能只生成一句自然语言回复，它还要结合当前战斗状态、角色属性、装备、历史剧情、规则判定和状态变化来决定结果。

所以这里更适合 Agent，而不是单纯的聊天模型。Agent 的价值在于：它不仅能生成文本，还能根据用户意图决定是否查规则、是否读状态、是否调用工具、是否推进游戏状态。

## 4. 为什么不直接用 ChatGPT / DeepSeek？

因为直接用通用聊天产品，业务闭环是不完整的。

它们擅长通用对话，但并不直接管理我自己的游戏状态、战斗状态、长期记忆和知识库，也不会天然支持我定义的工具链，比如规则检索、状态读取、战斗推进、异步任务追踪。

这个项目的重点不是“选一个会聊天的模型”，而是“围绕模型搭建一个完整的业务系统”。模型只是能力的一部分，真正难的是把模型和状态、规则、工具调用、异步执行链路结合起来。

## 5. 这个项目和普通聊天机器人有什么区别？

最大的区别是：**它不是只生成文本，而是在维护一个持续演化的游戏系统。**

普通聊天机器人通常只做两件事：读上下文、生成回复。但这个项目还要处理：

- 多会话隔离
- 用户身份与权限
- 游戏状态持久化
- 战斗状态推进
- 长期记忆维护
- RAG 规则/设定检索
- 异步消息任务状态跟踪
- 幂等与执行安全

所以它更像一个“带 Agent 能力的状态化业务系统”，而不是一个简单的 chat bot。

## 6. 用户输入一句话后，系统内部完整流程是什么？

以当前异步链路为例，完整流程是这样的：

1. 用户在某个 session 中发送一句话。
2. HTTP 层先做鉴权、参数校验和会话归属校验。
3. 系统读取当前 session，把这条用户消息追加到会话历史中。
4. 在一个 PostgreSQL 事务里同时写入：
   - 用户消息
   - `message_job`
   - `outbox_event`
5. 接口先返回 `202 Accepted`，并返回 `message_id / job_id`。
6. 后台 `OutboxDispatchLoop` 扫描 `outbox_events`，把任务发布到 RabbitMQ。
7. 后续 worker 消费这个 job，加载：
   - 最近消息
   - session memory
   - game state
   - encounter
   - rules/lore 检索结果
8. Agent Runtime 根据请求复杂度决定走简单路径还是 ReAct 路径，调用模型和工具生成回复。
9. 系统把 assistant reply 写回 `session_messages`，并显式记录：
   - `reply_to_message_id`
   - `source_job_id`
10. 最后把 `message_job` 状态更新为 `processing / completed / failed`，前端通过 `GET /messages/{id}` 查询结果。

一句话概括，就是：**先可靠接单，再异步分发，再由 Agent 读取状态和知识生成回复，最后把结果写回。**

## 7. 你在这个项目里主要负责什么？

我基本负责了这个项目从设计到落地的大部分核心工作，重点包括四块：

第一是**数据库和状态建模**，包括 `sessions`、`session_messages`、`game_states`、`encounters`、`session_memories`、`message_jobs`、`outbox_events` 等核心结构设计。

第二是**RAG 和 Agent 链路**，我做了 `rules/lore` 双知识库检索，把规则和设定接入 Agent Runtime。

第三是**异步消息架构**，我设计并实现了 `message_jobs + outbox_events + RabbitMQ` 的异步消息入口和状态追踪链路。

第四是**执行安全性**，包括幂等、显式 reply 关联、状态机收紧、锁续约和后续恢复机制的设计。

## 8. 这个项目最难的技术点是什么？

我认为最难的技术点是：**把 Agent 推理和业务状态系统结合起来，而且要保持可控。**

因为 DND 不是开放式聊天，它既要求模型有生成能力，又要求系统能稳定处理规则、状态和工具调用。也就是说，它不是“模型调用一下就结束”，而是要把检索、状态读取、规则判定、回复生成和状态更新串成一条完整链路。

这里最难的是找到平衡点：既不能让模型完全自由发挥，也不能把系统做成纯规则引擎。要让模型有表达能力，但关键状态和规则要由系统约束住。

## 9. 这个项目最大的工程难点是什么？

最大的工程难点是：**异步一致性和执行可靠性。**

一旦从同步请求切到异步链路，就会出现很多典型问题：

- 用户消息已经入库，但队列没发成功怎么办？
- 同一个任务被重复投递怎么办？
- worker 执行到一半崩了怎么办？
- assistant reply 被写了两次怎么办？
- 长时间 LLM 调用时锁过期怎么办？

所以我引入了：

- `Outbox` 解决写库和发 MQ 之间的一致性问题
- `message_jobs` 记录任务生命周期
- `reply_to_message_id / source_job_id` 加唯一约束实现幂等
- 锁续约和状态机控制来提升执行安全性

这部分本质上是后端工程问题，不是简单的模型调参问题。

## 10. 如果重新设计这个项目，你会怎么改？

如果重新设计，我会更早做三件事。

第一，**更早做请求分流和模型分层**。不是所有请求都应该走多轮 ReAct，很多状态查询、简单规则问答、简单剧情推进，完全可以走 `flash + 单次生成`，这样延迟会低很多。

第二，**更早把高层上下文工具做出来**。不要让 ReAct 一轮轮去查 history、memory、game state、rules、lore，而是先统一构建上下文包，减少 step 数和工具往返。

第三，**更早把完整 worker 闭环和恢复机制做完整**。如果从第一版就把独立 worker、stale processing recovery、published repair 这些链路做完整，异步系统会更稳定。

如果用一句话总结，就是：**我会更早把系统从“模型优先”调整成“路由优先、状态优先、执行链路优先”。**
