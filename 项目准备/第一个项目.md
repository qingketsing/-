1. 这个项目是什么？
	这个项目是一个分布式的，多节点的存储系统内核。这个项目既是技术验证，也是为了个人网盘而设计的。为的是解决大家在构建网络存储系统时由于要购买多个服务器又想要把所有的资源，特别是存储和处理器，能够整合在一起当作一个整体来使用。目前本项目正处于持续的更新中，仍然存在一些设计和细节问题。
2. 这个项目用了什么技术栈？
	这个项目是基于Golang开发的，使用的技术栈包括Redis，RabbitMQ，PostgreSQL和Docker。Redis负责处理查询热点信息和减少对数据库的压力，RabbitMQ通过消息队列异步处理信息，减小服务器的同步处理压力。PostgreSQL用于存储文件元信息，Docker是目前为了方便测试而引入的，后续会继续引入K8s来降低部署门槛，提高可用性。
3. 它和用 MinIO 直接搭一个对象存储有什么区别？
	MinIO 是一个纯粹的基础设施层对象存储引擎，而 AstraStorage 是一个在对象存储之上构建的、带有复杂元数据管理和强一致性调度的分布式文件管理系统。MinIO使用一种基于确定性哈希算法的去中心化架构进行数据放置，它内部有自己的分布式管理，但不依赖像 Raft 这种标准共识协议来管理元数据。而且MinIO的纯对象存储的查询能力比较弱（主要靠 prefix/后缀查询或借助对象标签），不适合做高频的、类似关系型表结构的文件列表检索。这个项目专门实现了底层的 TCP Server 用于上传和下载 (`tcp_server.go`)。这是为了规避 HTTP 协议的开销，实现更高效的流式传输、极速分片或原生的断点续传。
4. 如果当时用 etcd，你的开发周期能缩短多少？为什么没选这条路？
	如果引入 etcd，你的系统架构将变成：应用节点 + Postgres + Redis + RabbitMQ + etcd 集群。这会让部署变得比较臃肿，通过自己手写或内嵌 Raft，AstraStorage 的节点本身就是一个强一致性集群。这个项目的node跑起来，既是业务节点，又是共识节点。对内高内聚，对外只需要拉起 Node 即可。etcd 本质是一个强一致性的 KV 存储（Key-Value Store）。如果系统需要的不仅仅是存取 KV。同时我不得不承认，这个项目是有一点“造轮子”的技术执念与学习目的的。
5. 你提到了“5节点集群”，为什么是5，不是3也不是7？这个数字怎么来的？
	这里我要澄清一点，这个项目支持的不只是五节点，实际上，它可以支持非常多的节点，但是在这里之所以暂时使用五个节点来测试，是因为这个节点数量相较于3和7而言，更加合适。首先是我们所熟知的三节点，当我们对其中一个节点进行停止工作更新的时候，那么就只剩下两个节点了，如果这两个节点突然有一个节点出现网络抖动，那么这个服务就变的不可用了，至于为什么不使用7节点，则是出于在开发中方便维护和开发的想法， 节点越多，一次写操作所需的网络 RPC 广播就越重，这个项目的等待过程时间会显著变长。但是这并不意味着我的项目只能用5节点，这个系统是支持动态扩缩容的。
6. PostgreSQL 在这里是不是也成了单点？如果 PostgreSQL 挂了，系统还能用吗？
	在目前的基础部署拓扑结构中，是的，单个 PostgreSQL 实例确实是元数据层面的单点故障（SPOF）。我的核心精力目前集中在分布式系统的存储底座（自定义 RPC、Raft 强一致性集群、TCP 传输）上。PostgreSQL 作为外部组件，目前处于单机容器化部署状态，没有配置高可用集群。
	在设计这里的PostgreSQL的时候，我其实参考了k8s的边车容器设计理念，就是k8s的一个pod中既有工作的容器，也有用来处理日志和监控的边车容器，而PostgreSQL在这里就像是边车容器一样，处理文件的元信息。
7. 为什么是 RabbitMQ 而不是 Kafka？你对比过吗？
	Kafka更适合处理处理巨大的吞吐量和大数据日志流，仅仅为了解耦几个业务动作去拉起甚至维护一套 Kafka 集群过于沉重。自带完善的死信队列（Dead Letter Exchange）、延迟队列，以及精准的单条消息 ACK 机制。在这个项目中更为合适。如果 AstraStorage 未来演进出一个专门收集所有节点真实流量日志（Access Log）并进行实时数据分析的监控模块，面对每秒十万级的顺序写入，当然要引入 Kafka。但在当前阶段，我是用 MQ 来做**核心文件状态机的解耦与异步控制**，RabbitMQ 的精细度和路由能力是最佳选择。
8. 数据自愈的 selfHeal 机制具体怎么设计？是定期扫描全量数据对比吗？性能能接受吗？
	目前确实是使用的定期全量扫描，对于个人存储而言，内容数量可能并不大，相对而言体感上可以接受，但是不可否认的是，在大规模的企业存储中，这个确实不够看。我之前也考虑过这个问题，想到的解决办法是通过线性一致性来解决，这个在我的Raft中有所体现，只是现在还暂时没有嵌入到业务层面中，只是在Raft的部分有相关代码。在Raft中，我维护了一个commit-index的参数，当发送心跳的时候，会向所有子节点发送自己的这个参数，如果子节点发现自己的日志落后于leader，就会通过**AppendEntries RPC**来跟上leader，这样的话，初始一致，过程内容和顺序一致，由于满足线性一致性的要求，可以得出结果也是一致的。
9. 监控告警有吗？比如 Leader 挂了怎么通知运维？
	在 AstraStorage 当前的版本中，我们主要实现了应用层视角的系统指标采集。目前系统的监控大多停留在被动暴露（Pull-based）或打日志层面，缺乏一套主动推送给运维侧（钉钉/飞书/邮件）的闭环告警链路。我认为可以写一套关于这个系统的监控和消息推送系统，及时主动的通过邮件来联系运维。
10. **为什么选择Raft而不是Paxos？** 你理解两者的核心区别吗？
    首先就是方便理解和实现的程度上，Raft的理解难度和论文中的实现细节比Paxos要强很多，在项目中几乎可以复现Raft，毕竟Paxos以难以理解著称，Raft的设计目标就是可理解性。其次，Paxos的任何节点都可以发起提案，但是Raft只有leader可以，也就是说Leader 是强中心的，所有写请求必须经过 Leader。和这个项目更契合一些，因为这个项目的Redis和RabbitMQ都是和Leader进行绑定的。
11. **5节点集群，最多能容忍几个节点故障？为什么？**
    根据Raft需要超过半数的同意才能选举leader，所以这里的正常工作节点数量要大于5/2 +1，也就是最多两个节点故障。
12. **你说“基于下载速度的贪心策略选最速节点”，这个速度是怎么测的？会不会影响用户体验？**
    这里的速度是在心跳检测过程中进行的极短时间的网络测试，这里由于我现在还是本地检测，并没有实际拉入网络，用了一种类似Mock测试的办法
	具体代码就是 cluseter中的heartbeat.go的initNetStats函数中的(ΔbytesIO × 8) / Δt(秒) / 1e6。这个函数在每次心跳的时候都会进行一次。然后存到args中，通过心跳RPC的参数传给leader。实际上这个网络测试过程的时间非常短，可以忽略。
	这个设计在本地验证了可行性，未来部署到真实网络环境后，只需要把Mock替换成真实IO统计就行。
13. **MinIO是怎么嵌入的？**
    这部分是存在问题的，我引入 MinIO 是为了验证计算层和存储层解耦的可能性，让节点本身变成无状态的，文件统一存到对象存储。但后来我意识到，当前单节点 MinIO 反而成了新的单点故障，和引入 Raft 做高可用的初衷冲突了。真正合理的做法应该是用 MinIO 分布式集群，或者干脆回退到本地磁盘 + Raft 5选3副本复制，后者对这个场景反而更直接。所以这个项目里 MinIO 是作为可选后端存在的，不是核心路径。我最近也正在进行这部分的修改工作。
14. **你提到了“存算分离”，具体哪部分是存储、哪部分是计算？为什么要拆开？**
	存储是文件存储，用的是本地文件存储。
	计算则是：
	- **Raft 共识**：决定谁是 Leader、日志怎么复制
	- **Leader 服务**：处理上传/下载/删除请求，选节点，生成 token
	- **RabbitMQ 消费**：解析消息，协调流程
	- **PostgreSQL**：存文件元数据（文件名、大小、存在哪里）— 元数据很小，属于计算层的"账本"
15. **Raft选举时，如果三个节点同时发起投票，会发生什么？你的代码怎么处理这种情况？**
    首先三个节点都会投自己一票，然后这次选举会失败，由于随机时间间隔的选举超时，哪怕是极端情况，也一定会有选举超时时间不一致的情况，这时候就会有一个节点的任期要更高，从而能够正确的选出leader。这个其实是Raft论文中的内容，在这里我特地把超时时间的随机数部分设置的比在之前做分布式网络课程作业的测试中更长一些，为的就是能够尽可能避免极端情况的发生。
16. **你说“选举收敛时间150ms”，这是怎么测出来的？网络延迟考虑了吗？**
    这个是我在本地测试中跑出来的，具体测试代码在项目中，代码中没有处理网络延迟，因为现在还没有把所有的节点和服务放到K8s中，但是这个网络延迟我并没有忽略，只是现在为的是先让项目有一个雏形，所以暂时没有进行处理。
17. **Leader挂了之后，新Leader选出来之前，这段时间的写请求怎么处理？**
    RabbitMQ会定期检测有没有leader，这里的定期是500ms，如果这段时间检测到没有leader，那么就会卡在那里，只接收任务，但是不处理，直到检测到新的leader后才会异步发给leader，然后进行处理。
    这个500ms是经验值，我打算后续做成可配置的，适应不同场景的响应时间要求。
18. **你实现了“基于磁盘余量的动态权重调度”来实现5副本自动平衡。你怎么把权重调度和Raft结合起来的？**
    权重调度是根据服务来处理的，比如对于上传而言，leader会对剩余的内存容量来找出合适的三个节点，而下载部分，则是通过对所有下载速度进行比较，然后找出最快的一个建立tcp连接。
19. **节点重启后“自动从存活节点拉取缺失副本”，如果拉取过程中又有新写入，怎么保证数据一致？**
    首先这里我确实要承认，这部分也是存在问题的，对于新的数据写入，这部分是没有处理的，这个问题也确实存在。目前也确实在对这个进行修改，目前有一个selfHeal，就是通过心跳来保证数据统一，不过目前并没有完全实现。
20. **你设计“固定40字节二进制header”时，怎么解决TCP粘包问题的？**
    简历上写的是当时的实现，这之后我做了一些性能调研和协议对比。结论是：对 1MB 以上的大文件，三种协议吞吐量基本无差异，协议头解析开销可以忽略；但对 4KB 小文件高频传输，二进制头确实有优势，因为它用 io.ReadFull。一次读完固定长度，零内存分配，不需要逐字节扫描分隔符。**最终我决定保留文本协议**，因为这个项目的使用场景是个人服务器存储，小文件高频传输不是主要场景，而文本协议的**可维护性和调试便利性**更有价值。这个取舍让我理解了一件事：性能优化要先测量，不要凭感觉换方案。
21. **你说“小文件高频场景比HTTP提升54%”，这个测试是怎么做的？网络环境一样吗？**
    这个测试是在本地进行的测试，网络环境是一致的。
22. **你的二进制协议支持断点续传，客户端和服务端怎么协商断点位置的？**
	这个在我的项目的tcp传输过程中有一个参数是offset，可以找到断点前的偏移量，然后在这个偏移量之后进行分片重传
23. **分片MD5校验，如果校验失败，重传策略是什么？全部重传还是只传失败的分片？**
	断点续传是在上传服务的实际代码里实现的，走的是文本协议那条路：客户端重新发 metadata 请求，服务端通过文件名查询。找到上次中断的 offset 和临时文件，把 offset 回给客户端，客户端从那个位置继续发。两件事在不同层面，没有关系。
	断点续传只在**上传**方向实现了。客户端重新发 metadata 请求，服务端用文件名查内存里的 partialSessions，找到上次中断的 offset 和临时文件路径，把 offset 返回给客户端，客户端从那个位置继续发剩余数据。下载方向目前还没做断点续传，是个 TODO。
24. **元数据异步落库，如果异步写入失败，怎么保证数据最终一致性？**
    目前项目里 Leader 本地 DB 写入是同步的，写失败只打日志，没有重试或回滚。这是项目里一个已知的一致性漏洞，如果要修复，可以引入写入重试队列，或者把元数据广播也纳入 Raft log，通过 Raft 的一致性来保证元数据在所有节点上的最终落库。
